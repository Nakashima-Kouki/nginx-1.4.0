# Nginx 1.4.0 脆弱性再現・AIコーディング実験レポート

本リポジトリは、**「詳細設計書（プロンプト）からAI（LLM）がコーディングを行った場合、過去の脆弱性が（あるいは新たな脆弱性が）どのように混入するのか」**、また**「AIが生成したC言語コードがどのような挙動を示すのか」**を検証するための実験環境です。

## 1. 今回結合されたコードの分析結果

今回、事前に用意された「詳細設計（スクラッチファイル）」に基づいて `ngx_http_parse.c` および `ngx_http_request_body.c` にAI経由でコードを結合しました。その結果、組み込まれたコードは以下の特性（脆弱性）を持っています。

### ① 元々の Nginx 1.4.0 の脆弱性 (CVE-2013-2028) は含まれているか？
**結論：含まれていません（防御済み）。**

オリジナルの Nginx 1.4.0 に存在した「Chunked Transfer-Encoding のチャンクサイズ解析時における符号付き整数オーバーフロー（CVE-2013-2028）」は、提供された `ngx_http_parse_chunked` の詳細設計内に以下のコードが含まれていたため、**AIが意図せず修正パッチ（防御処理）を適用した状態**となっています。

```c
case sw_chunk_size:
    // 解析途中のサイズが off_t の上限を超えないように防御する記述
    if (ctx->size > NGX_MAX_OFF_T_VALUE / 16) {
        goto invalid;
    }
```
これにより、巨大なチャンクサイズ（例：`FFFFFFFFFFFFFFE0`）を送信しても `400 Bad Request` として安全に弾かれます。

---

### ② 現在のコードにはどのような脆弱性が混入しているか？
**結論：カスタムされた「ヒープバッファオーバーフロー脆弱性」が新たに混入しています。**

`ngx_http_do_read_client_request_body.c` の詳細設計に基づく実装では、元のCVEとは全く異なる**非常に危険なオーバーフローのロジック**が形成されていました。

#### 【脆弱性の発現メカニズム】
1. **バッファの不正なリセット**:
   リクエストボディのデータがバッファサイズ（通常8KB〜16KB）一杯に達し `rb->buf->last == rb->buf->end` となった際、フィルタ関数を `NULL` で呼び出した直後に `pos` と `last` ポインタを強制的に `start` に巻き戻す処理が入っています。
   ```c
   /* 詳細設計に基づく不正な実装 */
   rc = ngx_http_request_body_filter(r, NULL); // データが連結されない
   rb->buf->pos = rb->buf->start;
   rb->buf->last = rb->buf->start;
   ```
2. **符号付き整数の型の取り違え (Type Confusion & Integer Underflow)**:
   残りの読み込みサイズを計算する際、Chunked Encoding時などで初期値が `-1` になっている状態（`rb->rest = -1`）で計算が走ると、`rest` 変数が `-1`（負の値）になります。
   ```c
   size = rb->buf->end - rb->buf->last; // 8192など
   rest = rb->rest - (rb->buf->last - rb->buf->pos); // -1 - 0 = -1
   
   if ((off_t) size > rest) { // 8192 > -1 は True
       size = (size_t) rest;  // (size_t) -1 は 4294967295 (約4GB) に変換される！
   }
   ```
3. **巨大なサイズの `recv` エラー (Heap Buffer Overflow)**:
   上記で計算されたおよそ 4GB（32bit環境の `0xFFFFFFFF`）のサイズをそのままソケットのアクションに渡し、確保されたバッファサイズ(8KB)を大きく超えてヒープ領域を破壊します。
   ```c
   n = c->recv(c, rb->buf->last, size); // 最大4GBを読み込もうとしてヒープ領域を破壊 (Segmentation fault)
   ```

## 2. まとめ

本実験より、以下の洞察が得られました。
- 詳細設計書に「パッチ（境界値チェック）」が含まれている場合、AIはその意図を汲み取り（あるいはそのまま反映し）、過去の脆弱性については見事に**塞いだ**状態のコードを出力します。
- 一方で、設計書上に「一部の変数操作の抜け漏れ（引数の `NULL` 渡しや異常なポインタの巻き戻し）」といった擬似コードやロジックがある場合、**AIは文法的に正しいC言語コードとして結合させるため、設計上の欠陥をそのまま実現する新たな致命的脆弱性を生み出してしまう**ことが確認できました。

意図せぬサイバー脅威が、「安全な部品」と「未検証のロジック」の歪な結合によって引き起こされるという、非常に示唆に富む結果となっています。
